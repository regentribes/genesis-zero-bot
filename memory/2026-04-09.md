# 2026-04-09 — RegenTribes Digital Twin Session

## What we worked on

### big_space + Telegram Topic Visualization

**Goal:** Visualize the "digital twin" Telegram topic as a navigable 3D space where messages are entities.

**Stack decision:** Originally requested Bevy + big_space Rust WASM. Core big_space compiled to WASM fine, but Bevy rendering engine compilation timed out on this Hetzner VPS. Settled on Three.js/WebGL as delivery mechanism — same conceptual model, browser-native.

### Deliverables

1. **Scene data export** — Parsed full topic export (765 messages, Feb 20–Apr 9, 11 authors, 631 links)
   - `big_space_scene.json` — V1 coordinates (X=time, Y=author, Z=text length)
   - `big_space_scene_v2.json` — V2 coordinates (X=author, Y=theme cluster, Z=time, recent=close)

2. **Three.js visualization** deployed at `https://labs.regentribe.org/big_space_telegram/`
   - `telegram_bigspace/` — Full Three.js project (Vite build)
   - Label sprites appear on nearby nodes, word-wrapped text
   - Hover info panel, click-to-select with camera fly-to
   - Orbit/pan/zoom controls, author legend

3. **V2 coordinate system:**
   - X = author (11 people spread horizontally, large labeled spheres)
   - Y = theme cluster (60 clusters, messages grouped by shared link domains + temporal proximity)
   - Z = time (recent=close, oldest=far)
   - Lines to author = message authorship
   - Lines between nodes = reply chains

### nginx configuration
The nginx config at `/etc/nginx/sites-enabled/labs.regentribe.org` serves static files from `/var/www/html/`. The app must be built with `base: './'` (not `/big_space_telegram/`) for relative asset paths to work when deployed to a subdirectory.

Scene data files need to be copied explicitly to `/var/www/html/big_space_telegram/`:
```bash
sudo cp big_space_scene_v2.json /var/www/html/big_space_telegram/
```

### Key errors fixed
- "Failed to load scene data" — nginx wasn't serving JSON at root path. Fixed by adding explicit location block for `big_space_scene_v2.json`
- Vite build `base: '/big_space_telegram/'` → paths were absolute, broken when served from subdirectory. Fixed with `base: './'`
