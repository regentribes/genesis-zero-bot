## 11. CRDT STACK DEEP DIVE (2026-04-29)

### CRDT Fundamentals (from crdt.tech)
CRDT = Conflict-free Replicated Data Type. Optimistic replication — users may modify data on any replica independently, even if disconnected. Conflicts resolved automatically when replicas communicate. Supports decentralized operation — no single server required, works in P2P networks.

Two classes:
- CvRDT (Commutative): state-based, send delta to neighbors
- CmRDT (Operation-based): operation-based, broadcast operations

Key property: no coordinator, no consensus protocol needed. Any two Silk instances exchanging sync messages are guaranteed to converge to the same graph state.

### Loro — Rust + WASM CRDT Library
- **What:** CRDTs for collaborative JSON documents. Text, rich text, movable tree, list, LWW map. P2P sync, delta updates, time travel through history.
- **Strengths:** Clean API, multi-language (Rust/JS/Swift), active development, well-documented.
- **Weaknesses:** Application layer, no storage backend, no query engine.
- **Verdict:** Strong for document/knowledge layer on top of storage. Not a standalone database.

---

### redis-rust — Redis-compatible, Rust, CRDT Replication
- **What:** Research project (not production). Redis RESP2 wire protocol, actor-per-shard, CRDT-based multi-node replication (Anna KVS-style gossip + anti-entropy). Passes Redis Tcl test suite at 80-100% throughput.
- **Strengths:** Rust + CRDT + Redis client compatibility. Deterministic simulation testing with partition/loss injection — serious validation methodology.
- **Weaknesses:** Research-only, no persistence, no blocking ops, eventual consistency only.
- **Verdict:** Reference implementation for CRDT replication architecture, not for primary data layer.

---

### Silk-Graph — Merkle-CRDT Graph Engine ⭐ Most Relevant
- **What:** Replicated graph store with schema enforcement at write time. Merkle-DAG + CRDTs, no leader, no consensus, no coordinator. Typed property-graph ontology (RDFS-level class hierarchy). Python + Rust.
- **Strengths:** Directly addresses knowledge graph need. Schema enforcement = can't write invalid governance state. Convergence guaranteed, backed by semi-formal proof + automated invariant checks. Extracted from production DevOps platform (Shelob), not greenfield.
- **Weaknesses:** New, small community, no large-scale field deployment yet.
- **Verdict:** Highest alignment with the use case. Merkle-CRDT graph with schema enforcement maps directly to knowledge graph layer Integral needs. Invest.

---

### Eidetica — Rust P2P DB, Merkle-CRDT, iroh Transport
- **What:** Decentralized local-first database. Merkle-CRDTs as sync mechanism. Uses iroh for P2P transport. Multi-transport sync (HTTP + Iroh P2P simultaneously). WASM planned for web embedding.
- **Strengths:** Rust-native, P2P by design, privacy-first. iroh transport is serious infrastructure (used by significant projects). Multi-transport works in intermittent connectivity.
- **Weaknesses:** P2P connectivity still minimal. Storage format pre-1.0, no backwards compatibility guaranteed yet. Young project.
- **Verdict:** Worth monitoring. iroh transport layer solves the transport problem for mesh networking.

---

### Synthesist — Nomograph Claim-Based Spec Manager, CRDT Log
- **What:** Specification graph manager for AI-augmented collaborative development. Claims stored over append-only CRDT log. Multi-user collaboration via automatic CRDT merge. Full supersession history preserved per field.
- **Strengths:** Exactly the problem Integral faces — AI agents + humans collaborating on governance specs with proper versioning. Claim-based model maps to governance decisions and their evolution.
- **Weaknesses:** Niche tool for AI-augmented development, not a general database.
- **Verdict:** Reference model for how CRDTs can track governance claims and their evolution. Not a production component.

---

### Arxia — Offline-First Blockchain over LoRa/BLE/SMS/Satellite ⭐ Directly Relevant
- **What:** Layer 1 blockchain for zero-internet environments. Block-lattice (per-account DAG chains). CRDT state reconciliation (PN-Counter, OR-Set, Vector Clocks). Open Representative Voting (ORV) consensus. Transport: LoRa, BLE, SMS, satellite.
- **Strengths:** This is exactly what the infrastructure layer needs. Offline payment capability. Mesh radio transport. CRDT reconciliation for partition tolerance. Rust implementation. Block-lattice avoids heavy consensus overhead of traditional blockchains.
- **Weaknesses:** Very early (no mainnet, no real-world stress testing). ORV consensus has different security assumptions than PoW/PoS — needs scrutiny under adversarial conditions.
- **Verdict:** Highest priority for infrastructure layer. Directly solves mesh networking + offline operation. Needs deeper evaluation and test deployment.

---

### AFFiNE — Privacy-First Local-First Knowledge Base
- **What:** Notion + Miro replacement. Docs, canvas, tables fused. Local-first, open-source, self-hostable.
- **Strengths:** Production-ready, polished UI, self-hostable. Community-facing knowledge interface.
- **Weaknesses:** Not designed for P2P sync or sovereign deployment out of the box. Collaboration features require central server.
- **Verdict:** Good for human-facing layer if self-hosted. Not a CRDT sync solution.

---

### BlackSwan — TRL-4 Autonomous AI Swarm, TLA+ Verified
- **What:** Autonomous self-improving AI swarm. Formal TLA+ specs (NodeLifecycle, D2BFT, GlobalState, SporeProtocol, Ouroboros). Multi-agent economic simulator. Docker cluster with self-healing.
- **Strengths:** Formal verification at TRL-4 is serious. Ouroboros distributed evolution, self-sovereign economy with Kelly criterion capital allocation, defense in depth, swarm resilience via Spore Protocol.
- **Weaknesses:** Russian-language project, high complexity, economic model may not align with Integral's non-monetary ITC design. Docker-based implies infrastructure not available offline.
- **Verdict:** Reference for formal verification methodology and swarm resilience. Not a production component.
