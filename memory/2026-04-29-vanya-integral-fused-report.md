## Part II: The Integral Closed-Loop Design Framework

### Core Loop

```
CDS → OAD → COS → ITC → FRS → CDS
```

### Five Subsystems

**CDS (Community Deliberation System)**
- Authorizes projects and defines boundaries
- Sets normative boundaries
- Monitors for mission creep
- Restricted to aggregate data, quarterly review

**OAD (Outcome Allocation & Design)**
- Searches global repository, adapts to local context
- Calculates ecological footprint
- Estimates labor and materials
- Safeguards: simulation and peer review before authorization

**COS (Coordination & Operations System)**
- Decomposes tasks, matches skills to jobs
- Schedules workflow, reserves materials
- Tracks progress
- Safeguards: role rotation, FRS pattern monitoring

**ITC (Input-Output Tracking)**
- Tracks contribution balances
- Multi-source verification
- Safeguards against gaming

**FRS (Feedback & Response System)**
- Monitors outcome gaps
- Routes signals to CDS for deliberation
- Safeguards: restricted to aggregate data, anomaly detection

### Failure Modes

| Failure | Safeguard |
|---|---|
| Subsystem capture (one dominates) | CDS sets normative boundaries |
| Deliberation paralysis | Consensus gradient, timeboxing, Syntegrity |
| OAD stagnation (untested designs) | Simulation and peer review before authorization |
| COS hierarchy emergence | Role rotation, FRS pattern monitoring |
| ITC gaming | Multi-source verification, FRS anomaly detection |
| FRS surveillance creep | Aggregate data only, quarterly CDS review |

### Implementation Phases

**Phase 1 (Months 1-6):** Paper-based
- CDS-lite: deliberation on paper
- OAD: collecting designs
- ITC: piece of paper tracking contributions
- COS: shared calendar, tool sign-out
- FRS: basic monitoring

**Phase 2 (Months 6-18):** Community-controlled digital
- Server infrastructure
- Digital CDS deliberation platform
- ITC ledger with contribution recording
- OAD design repository with version control
- COS coordination tools with task tracking
- FRS monitoring dashboards

**Phase 3 (Months 18-36):** Data-connected
- All five systems live and data-connected
- Automated feedback routing
- Cross-node federation (minimum 2 other nodes)
- Advanced OAD simulations and ecological calculators
- ITC recalibration mechanisms
- FRS anomaly detection trained on local patterns

**Phase 4 (Year 3+):**
- Full stress testing of all failure modes
- Physical backup for critical digital functions
- Community members can operate manually if digital fails
- Inter-node trade and knowledge exchange active
- Community contributes designs to global OAD commons

## Part V: The Exact Method

### The Core Mechanism

**An integral node is not a cost center. It is a revenue-generating service that happens to also be infrastructure.**

The trick: **build the node *as* the service, not *before* the service.**

### Phase 1: The Survival Job (Week 1-4)

**Pick one physical system you can observe and one person who manages it.**

Not a platform. Not a toolkit. One system. One client.

**Vanya's JTBD洗干净 (stripped of guru packaging):**

```
TRIGGER: What event makes the person act?
CURRENT PROCESS: How do they solve it now?
SUCCESS CRITERIA: How do they know it's solved?
EMOTIONAL COST: What does failure cost them?
```

**The actual question:** What does this person hire you to do, what will they pay for, and what is the minimum thing that does it?

**Example:** Neighbor has a greenhouse. They manually check soil moisture twice a day, worry constantly, have killed plants from over/under watering.

- Trigger: Morning anxiety about whether to water
- Current process: Finger test, guess, overcompensate
- Success criteria: Plants thrive, zero anxiety, no dead crops
- Emotional cost: Financial loss + gardener shame

**Build:** Telegram bot. Soil moisture sensor sends reading every 4 hours. Bot says "water today" or "wait." No app. No dashboard. No causal model.

**Charge:** $50/month. Or barter. Or nothing upfront, take 10% of saved input costs. Align with business outcome, not time.

**Result:** Income stream + real-world validation + one functioning sensor system you now own.

### Phase 2: The Node Seed (Month 2-3)

**Use Phase 1 income and infrastructure to add one subsystem.**

**OAD-lite:** Document system specs: sensor model, agent logic, calibration thresholds, failure modes. Shared folder. Spreadsheet. Paper binder.

**ITC-lite:** Track hours spent maintaining, cost of materials, response time when something broke. Contribution ledger.

**What you're building:** The *evidence* that this works. Not the full integral node. The proof that the node concept generates income.

### Phase 3: Replicate (Month 3-6)

**Find 2 more clients with similar physical systems.**

Same job, same sensor, same bot, same pricing model.

Each additional client:
- Covers survival costs
- Generates surplus for node infrastructure
- Provides data for OAD design repository
- Builds ITC contribution ledger across multiple members

**By month 6:**
- 3 paying clients (survival income covered)
- 3 functioning sensor-agent systems (OAD seed)
- 3 data streams on usage (ITC seed)
- Shared decision record on what works (CDS seed)
- Shared resource tracking (FRS seed)

### Phase 4: Actual Integral Node Construction (Month 6-18)

**Only now — digitize and connect.**

**CDS-lite:** Shared document. Agreements about what we're building, who decides, what we won't do. Monthly 30-minute call.

**OAD-lite:** Folder with 3 functioning system specs. When 4th client comes with different system, adapt from existing specs.

**COS-lite:** Shared calendar. Tool sign-out sheet. Who's available to help install. Not a task management platform.

**ITC-lite:** Spreadsheet. Client name, hours, materials cost, payment. Quarterly review: contributions balanced? Anyone exploited?

**FRS-lite:** Same spreadsheet. Monthly: income covering costs + node investment? Outcome gaps?

### Phase 5: Federation (Year 2+)

**Connect with one other node.**

- Share OAD designs (they have greenhouse systems, you have water tank systems — now both have both)
- Share ITC patterns
- Share FRS monitoring
- Cross-train on COS coordination

**Inter-node trade is not just money.** It's knowledge, labor, and designs.

## Part IX: The One Condition

**You must be physically present in the community you're serving.**

Vanya's "work with any business in the world online" is correct for purely digital services. The integral node — by definition — requires physical systems, physical installation, physical trust, physical maintenance.

**This is a local economy method disguised as a tech method.** It works if you're in the community. It doesn't work if you're remote.

**Implication:** The people who will build integral nodes are not gig-economy workers serving global clients from a laptop. They are neighbors who install sensors in each other's greenhouses, who meet monthly to decide what they're building together, who show up when a system fails.

Vanya optimizes for individual career survival. The integral node optimizes for collective resilience. Both can be true simultaneously — but only if you choose the community as your market, not the global internet.
